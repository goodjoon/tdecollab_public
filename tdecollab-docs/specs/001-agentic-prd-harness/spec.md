## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 시나리오 1 - 웹 기반 AI 대화를 통한 기획 문서 작성 및 Confluence 업로드 (Priority: P1)

기획자는 웹 시스템을 통해 AI 에이전트와 대화하며 기획 아이디어를 구체화합니다. AI는 사용자의 입력에 반응하여 PRD, 기능정의서, 테스트 계획서 등의 초안을 작성하고 항목별로 수정할 수 있는 인터페이스를 제공합니다. 작성이 완료된 문서는 사내 Confluence 페이지로 업로드됩니다.

**우선순위 이유**: 본 시스템의 가장 핵심적인 기능으로 기획 단계의 효율성을 극대화하기 위해 가장 먼저 확보되어야 합니다.

**독립적 테스트**: AI와의 대화를 통해 텍스트 입력을 받아 구조화된 기획 문서 항목들을 웹 상에서 조회 및 수정할 수 있고, 'Confluence에 업로드' 기능을 통해 실제 사내 Confluence에 페이지가 생성되는지 독립적으로 검증할 수 있습니다.

**인수 시나리오 (Acceptance Scenarios)**:
1. **Given** 기획자가 웹 화면에 접속하여 초기 아이디어를 입력하면, **When** AI 에이전트가 이를 분석하여 PRD 및 기능정의서 초안을 항목별로 생성하면, **Then** 기획자는 각 항목을 웹 화면에서 직접 수정하거나 AI에게 수정을 요청할 수 있다.
2. **Given** 작성 완료된 PRD 항목들이 있을 때, **When** 'Confluence 동기화' 액션을 실행하면, **Then** 사내 Confluence에 정해진 템플릿에 맞게 페이지가 생성되거나 업데이트된다.

---

### 사용자 시나리오 2 - JIRA 개발 Task 자동 분할 및 발행 (Priority: P1)

Confluence에 기획 문서가 확정되면, 시스템은 PRD 및 기능정의서 내용을 바탕으로 개발 Task들을 도출합니다. 각 Task 간의 선후 관계를 고려하여 우선순위 및 Task 번호를 생성하며, Task 내용에는 관련 PRD 링크, 기능정의 내용, 구현 방안 및 설계 정보가 포함되어 JIRA 티켓으로 자동 발행됩니다.

**우선순위 이유**: 기획 문서에서 실제 개발 워크플로우로 자연스럽게 이어지는 자동화의 핵심 단계입니다.

**독립적 테스트**: 확정된 기획 문서 데이터를 입력으로 제공했을 때, JIRA 서버에 설계 정보와 의존성(선후 관계)이 포함된 올바른 형태의 티켓들이 생성되는지 검증할 수 있습니다.

**인수 시나리오 (Acceptance Scenarios)**:
1. **Given** 확정된 PRD와 기능정의서가 시스템에 로드되어 있을 때, **When** '개발 Task 생성'을 요청하면, **Then** 시스템이 구현 방안 및 설계가 포함된 JIRA 티켓 초안 목록을 보여준다.
2. **Given** 생성된 Task 목록이 있을 때, **When** 'JIRA에 발행'을 실행하면, **Then** 사내 JIRA에 Task 간 선후 관계(의존성)가 설정된 상태로 티켓들이 생성된다.

---

### 사용자 시나리오 3 - GitLab 연동을 통한 JIRA 티켓 상태 자동 업데이트 (Priority: P2)

개발자가 발행된 JIRA 티켓을 참고하여 개발을 진행한 후, GitLab에 티켓 번호를 멘션하여 커밋 및 푸시를 수행하면, 해당 JIRA 티켓의 상태가 자동으로 변경됩니다. MR(Merge Request)을 생성하면 JIRA 티켓 상태가 리뷰(Review) 단계로 전환됩니다.

**우선순위 이유**: 개발자의 작업 편의성을 높여주나, 1, 2번 시나리오 구축 후 도입 가능한 부가적인 자동화 기능입니다.

**독립적 테스트**: JIRA 티켓이 존재할 때 GitLab에서 커밋과 MR 이벤트를 발생시키고 JIRA의 티켓 상태가 약속된 상태로 올바르게 전이되는지 모의(Mock) 이벤트를 통해 검증할 수 있습니다.

**인수 시나리오 (Acceptance Scenarios)**:
1. **Given** 개발자가 대상 JIRA 티켓 번호를 포함하여 커밋/푸시하면, **When** GitLab 시스템과 연동된 모듈이 이를 감지하면, **Then** JIRA 티켓의 상태가 알맞게 변경된다.
2. **Given** GitLab에서 MR이 생성되었을 때, **When** 연동 모듈이 MR 제목이나 내용에 있는 JIRA 티켓 번호를 확인하면, **Then** 대상 JIRA 티켓의 상태가 코드 리뷰 단계로 업데이트된다.

## 요구사항 *(필수)*

### 기능 요구사항 (Functional Requirements)

- **FR-001**: 시스템은 기획자와 대화하고 문서 초안을 생성할 수 있는 웹 기반의 채팅 및 문서 편집 인터페이스를 제공해야 한다.
- **FR-002**: 시스템은 기획자의 요청에 따라 PRD, 기능정의서, 단위/통합 테스트 계획서를 항목별로 생성 및 수정할 수 있어야 한다.
- **FR-003**: 시스템은 생성된 기획 문서들을 사내 On-Premise Confluence API를 사용하여 페이지로 게시 및 업데이트해야 한다.
- **FR-004**: 시스템은 확정된 기획 문서를 분석하여 개발 Task를 도출하고, 선후 관계 및 우선순위를 자동으로 부여해야 한다.
- **FR-005**: 시스템은 각 개발 Task에 PRD, 기능정의, 구현 방안 및 설계 내용을 포함하여 사내 On-Premise JIRA에 티켓을 발행해야 한다.
- **FR-006**: 시스템은 GitLab의 이벤트를 감지하거나 GitLab 파이프라인과 연동하여 커밋/푸시 및 MR 시 연관된 JIRA 티켓 상태를 자동으로 업데이트해야 한다.
- **FR-007**: 시스템 내부의 AI 도구(하네스)는 사내 Confluence, JIRA, GitLab과 상호작용하기 위해 MCP(Model Context Protocol) 래퍼(Wrapper) 서브 프로젝트(`mcp/`, `confluence/`, `jira/`, `gitlab/`)를 활용해야 한다.

### 주요 엔티티 (Key Entities)

- **기획 문서 (Specification Document)**: PRD, 기능정의서, 테스트 계획서 등의 정보 객체. (항목별 수정 및 Confluence 페이지 연동 속성 포함)
- **개발 Task (Development Task)**: 구현을 위한 단일 작업 단위. (JIRA 이슈 매핑 정보, 구현 방안, 설계 정보, 선후 관계, 우선순위 포함)

## 성공 기준 (Success Criteria) *(필수)*

### 측정 가능한 결과

- **SC-001**: 기획자가 최초 아이디어 입력부터 PRD 및 기능정의서를 작성하고 Confluence에 업로드하기까지의 모든 과정을 웹 환경 내에서 완수할 수 있다.
- **SC-002**: 기획 문서 기반으로 생성된 JIRA 티켓은 모두 누락 없이 개발 순서(선후 관계)와 명확한 구현 방안을 포함한다.
- **SC-003**: GitLab에서 커밋 및 MR 발생 시, 별도의 수작업 없이 JIRA 티켓 상태가 자동으로 업데이트된다.

## 가정 사항 (Assumptions)

- **환경**: 대상 시스템인 Confluence, JIRA, GitLab은 모두 사내 On-Premise 환경에서 정상적으로 운영되고 있으며 필요한 API 권한이 제공된다고 가정합니다.
- **아키텍처**: MCP Wrapper 역할을 하는 서브 프로젝트(`mcp/`, `confluence/`, `jira/`, `gitlab/`)와 웹 UI 프로젝트(`frontend/`, `backend/`)는 별도의 서브 모듈로서 서로 독립적으로 통신 가능한 구조라고 가정합니다.
- **GitLab-JIRA 연동**: 기존 GitLab-JIRA 연동 플러그인이 없거나 커스텀 로직이 필요한 상태이므로, 본 시스템에서 직접 상태 제어를 수행한다고 가정합니다.
